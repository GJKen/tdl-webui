# 上传用法
## 注意：没装 tdl 二进制，命令前缀是 go run . 而不是 tdl
（想直接用 tdl 命令就先构建：go build -o tdl.exe . ，之后用 .\tdl.exe up ...）

## 发到 Saved Messages（发给自己，最安全的测试）
go run . up -p "文件或目录路径"

## 发到指定群组 / 频道
go run . up -p "文件" -c 目标
- 目标可以是【用户名】(公开频道/群，如 -c durov) 或【数字ID】(私有的)。
- 数字ID 从 go run . chat ls 里查（表格里的 ID 是 raw id，不带 -100 前缀）。
- 私有目标第一次用前先跑一遍 chat ls，把 peer 缓存下来，否则按 id 解析不到。
- 话题群再加 --topic 话题ID（话题ID 也在 chat ls 的 Topics 列里）。
- 要对目标有发送权限（频道要管理员发帖权，群要在群里且能发媒体）。

## 查看可上传的目标
go run . chat ls                                  # 全部会话
go run . chat ls -f "Type == 'channel'"           # 只看频道
go run . chat ls -f "Type == 'group'"             # 只看群组

## caption（上传后消息底下的说明文字）
默认【不是空的】！不指定时会自动带上「文件名(去扩展名) - MIME类型」，渲染成等宽代码样式，例如：
  [AI-Raws] 天空戦記シュラト #01 (BD HEVC 1440x1080 FLAC)[45ABB24A] - video/x-matroska
--caption 收的是【expr 表达式】不是纯文本，注意引号嵌套（外层单引号给 PowerShell，内层双引号给 expr 当字符串）：
- 不要说明：     --caption '""'          ← 见下方踩坑，PowerShell 下只能这样写
- 只要文件名：   --caption 'FileName'
- 文件名带后缀： --caption 'FileName + FileExt'
- 固定文本：     --caption '"我的说明"'
可用字段：FilePath / FileName(无扩展名) / FileExt / ThumbPath / MIME
看全部字段：     go run . up --caption -

## 踩过的坑
- 【-c 前面一定要有空格】。写成 "xxx.mkv"-c 1234 会被 PowerShell 拼成一个参数，
  路径变成 xxx.mkv-c、-c 失效。正确写法：... .mkv" -c 1234
- 路径带空格 / 特殊字符([ ] # ! ( )) 要整体用双引号括起来。
  tdl 用 WalkDir 遍历、不做通配符展开，brackets 按字面处理，没问题。
- 【清空 caption 只能写 --caption '""'】(单引号里包两个双引号)。
  PowerShell 5.1 会把空参数直接吞掉：--caption "" 和 --caption '' 都会让 tdl 报
  "flag needs an argument: --caption"。而 '""' 会被当成空字符串传进去，正好得到空 caption。
- 单文件上限：普通号 2GB，Premium 4GB。
- 示例（把 ~1GB 的 mkv 发到 id 为 1202784229 的频道）：
  go run . up -p "D:\!Amine\!Watching\[AI-Raws][Shurato]\[AI-Raws] 天空戦記シュラト #01 (BD HEVC 1440x1080 FLAC)[45ABB24A].mkv" -c 1202784229 --caption '""'

### 混合相册冲突修复（2026-06-16）
- 【根因】相册模式服务端把每个文件塞进一个 message.Album(...) 一组发出。Telegram 不允许同一媒体组里混放「照片/视频」和「普通文件(文档)」，所以「图片 + 不能进相册的文件」必然在 album.go:145 报 send album 错。
- 【检测】前端把文件分三类——视觉(图/视频)、音频、文档；跨类即判为冲突（通用，不只图片+文件，音频混视觉、音频混文档也触发）。
- 【方案：勾选框仅在混合冲突时出现】
  - 不冲突：照常显示「作为相册」勾选。
  - 冲突：隐藏「作为相册」，改显示【全部作为文件发送（合并成一个相册）；取消勾选则分开逐个发送】，默认勾上。
  - 勾上 → 走相册接口并带 as_file=true：服务端每个文件 AsFile:true / Photo:false，当纯文档上传 → 全文档相册对 Telegram 合法（仍按 ≤10 自动分组）。
  - 取消 → 落回原来的逐个发送循环。
  - 增删文件时实时重判冲突。

  - app/web/upload.go — handleUploadAlbum 新增读取 as_file 字段，为真时 AsFile:true / Photo:false。

### 跨账号会话记录一致 + 无权限发送提示（2026-06-16）
1.【私聊记录双向同步】以前 A 发文件给 B（私聊），记录只挂在 A 名下；切到 B 打开和 A 的对话是空的。
   - 根因：上传任务用【目标】的 id 当 chat_id（A→B 时 chat_id=B），而 B 视角里这条会话是用【对方=A】的 id 标识的，两边永远对不上。

## 注意
- 只监听 127.0.0.1、API 无鉴权，别对外开端口。
- 同一时间别再开别的 tdl 进程：bolt 数据库是独占锁，会冲突。
- 登录仍只支持验证码方式；QR / tdata 导入、下载/转发还没做。【网页内实时上传百分比已做】（进度圆环）。【文件组/相册已做】：勾“作为相册”把多文件发成一条媒体组（≤10/组，超出自动分组）。
- 任务是【内存态】：重启 server 后“聊天记录”清空。头像颜色 / 置顶存在浏览器 localStorage（换浏览器或清缓存会重置）。
- 拖拽/选择上传会先把字节存到系统临时目录（%TEMP%\tdl-web-upload）暂存、发完自动清理；超大文件建议走“服务器本地路径”模式省一次传输。
- 想用 tdl server 而不是 go run：先 go build -o tdl.exe . ，之后 .\tdl.exe server。
