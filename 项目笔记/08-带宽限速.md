# 08 · 全局带宽限速 --rate（2026-06-19 已实现）

全局 flag `--rate`，下载/上传共用一个限速值，所有任务+所有分片共享一个 `rate.Limiter`（nil=不限速直通）。网页端后来加了独立设置弹窗：目前只管上传限速，下载限速项仍禁用占位。三个 module（root/core/extension）均 build + vet 通过。

## 用法
- `go run . dl -u xxx --rate 10M`、`go run . up -p xxx --rate 5M`
- 单位 1024 进制：`10M / 500K / 2.5MB / 1.5 GiB`；留空=不限速。
- `tdl server --rate 5M` 可作为网页端初始上传限速；但网页设置里保存过上传限速后，持久值优先于启动 flag。
- 网页端在会话列表左下角 ⚙ 设置上传速率，保存后会热更新，正在上传的任务也会跟随新速率变化。

## 关键代码位置
- `pkg/utils/byte.go` `Byte.Parse`（解析）、`pkg/utils/ratelimit.go` `NewRateLimiter`（burst=max(rate,1MiB)，<=0 返回 nil）
- `pkg/consts/flag.go` `FlagRate`、`cmd/root.go` 注册 PersistentFlags
- 下载单点：`core/downloader/progress.go` `writeAt.WriteAt` 里 `WaitN(ctx,len(p))`（ctx 已存进 struct，`newWriteAt` 传入）；`Options.Limiter`
- 上传单点：`core/uploader/uploader.go` `BuildMedia`（逐文件+相册都过这里）把 reader 包一层限速；`Options.Limiter`
- 接线：`app/dl/dl.go`、`app/up/up.go`、`app/up/album.go`（相册路径单独传 limiter）
- web 接线：CLI 路径不传 `up.Options.Limiter`，仍按 flag 现造 limiter；web 路径传入同一个共享 limiter 指针，覆盖 flag 路径，设置弹窗用 `SetRateLimit` 原地调整，所以不用重启 server，也能影响正在跑的上传。

## 仍需注意的坑
1. 【burst 下限，最关键】`burst=max(rate,1MiB)`，绝不能 `burst=rate`。下载分片 1MB，`WaitN(n>burst)` 会立即报错不阻塞，`--rate 500K` 会让第一个分片就挂。已在 `NewRateLimiter` 里处理。
2. 【core 是独立 module】`github.com/iyear/tdl/core` 不能 import root 的 `pkg/utils`（否则模块循环）。所以限速 reader 写在 `core/uploader` 内部，只把 `*rate.Limiter` 跨边界传。`core/go.mod` 已补 `golang.org/x/time/rate v0.15.0`（原本只有 root 有）。
3. 【nil 直通】`--rate` 没设时返回原始 file，行为与改前逐字节一致、零风险。（注：当前 gotd v0.140.0 上传本就是单协程串行读 `syncio.NewReader`，并行的只是网络发送端，不用 ReaderAt，所以包不包都不损失上传并行度——但 nil 直通仍保留。）
4. 下载是写侧限速，gotd 预取会有几 MB 启动突发、稳态准确；Resume 不受影响。
5. 【web 热更新必须复用同一指针】网页端不限速不能用 nil 表示，而是用 `rate.Inf` 的非 nil limiter；否则每个上传拿到不同 limiter，设置弹窗就只能影响新任务，不能影响进行中的任务。
6. 【web 目前只做上传】设置弹窗里的下载限速只是占位禁用；真正下载路径仍只受命令行 `--rate` 影响。
